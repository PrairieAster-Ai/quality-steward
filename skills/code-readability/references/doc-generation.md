# Doc generation (hybrid) + Wiki publishing

The `generate`/`publish` modes combine a **deterministic tool layer** (props tables, signatures — reproducible, can't drift from code) with a **prose layer** the model writes (overviews, examples, cross-links). Both read the same TSDoc.

## One-time setup

The doc tools are not project dependencies (they'd bloat installs). Install them on first run as devDeps (or run via `npx --yes`). In a monorepo, target the package that owns your components with `--workspace=<pkg>`; in a single-package repo, drop that flag:

```bash
npm i -D react-docgen-typescript
npm i -D typedoc typedoc-plugin-markdown
```

`react-docgen-typescript` is a library (no CLI) — it's driven by `scripts/extract-docs.mjs`. `typedoc` has a CLI.

## Tool layer

### Components → props tables

```bash
# Pass your own component globs; --tsconfig defaults to $CR_TSCONFIG or tsconfig.json.
node .claude/skills/code-readability/scripts/extract-docs.mjs \
  "src/components/**/*.tsx" "src/pages/**/*.tsx" \
  > /tmp/cr-docs/components.json
```

Emits one record per exported component: `{ filePath, displayName, description, props: [{ name, type, required, defaultValue, description }], tags }`. `description`/`props[].description` come straight from the TSDoc — which is exactly why the comments must be good first (run `annotate` before `generate` on low-coverage areas).

### Hooks / lib / api / database → Markdown

```bash
# Point --tsconfig / --entryPoints at your own source dirs. In a monorepo run
# one invocation per package; in a single-package repo, one is enough.
npx typedoc \
  --plugin typedoc-plugin-markdown \
  --tsconfig "${CR_TSCONFIG:-tsconfig.json}" \
  --entryPoints src/hooks src/lib \
  --out /tmp/cr-docs/api --readme none --githubPages false
```

For a Drizzle schema, **`scripts/gen-schema-page.mjs`** generates the whole `Reference-Database-Schema` page directly from your schema source (table summaries from each `pgTable`'s TSDoc; columns — type, PK/FK/not-null/default + inline rationale — parsed from the source). This reads far better than TypeDoc's type dump. The schema path defaults to `$CR_SCHEMA` (or `src/db/schema.ts`); the ER diagram and AI-context callout in that script are a generic template you customize once (see the `// TODO` notes in the script). Run: `node .claude/skills/code-readability/scripts/gen-schema-page.mjs <wiki-dir>`.

## Prose layer (the model)

For each area, wrap the tool output with narrative the tools can't produce:

1. **Overview** — 2–4 sentences: what the area is for, the mental model, how the pieces compose. Distil from the module headers + `CLAUDE.md`/`AGENTS.md`.
2. **Examples from real call sites** — `grep` the repo for actual usages; never invent. e.g. for a component, show how a page mounts it.
3. **Cross-links** — `[[Reference-Hooks#useMyHook]]`, links to your Architecture concepts (the domain terms you configured for the project).
4. **AI-context callout** — a short blockquote of the 2–3 invariants/gotchas a model most needs.

Page template (the names below are an illustrative example from the reference project — substitute your own area, components, and concepts):

```markdown
<!-- generated by /code-readability — edits will be overwritten -->
# Reference: Components — Prep

**Jump to:** [Overview](#overview) · [PrepLeafItem](#prepleafitem) · …   <!-- manual anchor-link TOC; GitHub wiki ignores [[_TOC_]] -->

> **AI context:** prep/cook share these renderers via props; quantities use the
> fraction formatter; stock-check-only items are excluded upstream.
<!-- plain blockquote on purpose — GitHub `> [!NOTE]` alerts don't render reliably in wikis -->

## Overview
<prose>

## `PrepLeafItem`
<TSDoc description>

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `leaf` | `PrepLeaf` | — | … |   <!-- from extract-docs.mjs -->

**Example**
```tsx
<PrepLeafItem leaf={leaf} showStorage showPrepInstructions />
```
See also: [[Reference-Hooks#usePrepDetail]], [[Architecture#Computed-views]].
```

## GitHub Wiki Markdown conventions (researched)

GitHub Wikis run on **Gollum** and use GFM, but differ from README markdown — generate for those differences, not against them.

### Structure follows Diátaxis

Auto-generated API docs are **Reference** material in the [Diátaxis](https://diataxis.fr/) sense: neutral, complete, and structured to *mirror the code* ("like a map reflects the territory" — facts, free of interpretation). Keep the four doc kinds in separate pages; mixing tutorial prose into reference is the most common documentation anti-pattern, and the generator must not do it:

- **Reference** (generated) — `Reference-*` pages: facts only.
- **Explanation** — `Architecture.md`: the *why* (your core domain model and key concepts). Don't bury this inside reference pages.
- **Tutorials / How-to** — hand-authored, never generated; the skill leaves them alone (marker check).

### Wiki mechanics that differ from READMEs

| Feature | Rule |
|---|---|
| Internal links | `[[Page]]`, `[[Page#heading-anchor]]` — **not** repo-relative `.md` paths. No `@mention` / `#issue` autolinks in wikis. |
| Wiki links **inside tables** | A *piped* wiki link `[[Page\|label]]` breaks a Markdown table cell — its pipe is parsed as the column delimiter, so the label garbles. Inside tables use a label-less `[[Page]]` or a standard link `[label](Page-Name)`; reserve `[[Page\|label]]` for prose. |
| Page naming | Both **spaces and `/` collapse to `-`** in the URL slug, so `Reference Components` and `Reference/Components` link identically. Name files with explicit hyphens (`Reference-Components-Prep.md`) and avoid spaces/slashes so links are unambiguous. |
| Navigation | Hand-build `_Sidebar.md` — the default sidebar is alphabetical and unhelpful. `_Footer.md` for a common footer. Both apply to every page. |
| Table of contents | **GitHub's hosted wiki does NOT render the Gollum `[[_TOC_]]` macro** — it shows as literal `[[_TOC_]]` text (verified on this repo's wiki, 2026-06). For long pages emit a **manual** TOC instead: a one-line `**Jump to:** [Heading](#slug) · …` list (GitHub auto-generates the lowercase-hyphenated heading anchors). |
| Heading anchors | GitHub slugs headings via `github-slugger`: lowercase, **keep word chars (incl. `_`) + hyphens**, spaces→`-`, drop other punctuation. So `### MyComponent` → `#mycomponent` **but `### my_table` → `#my_table`** (the underscore stays). Generators MUST build `[text](#anchor)` links with `scripts/wiki-slug.mjs` (`ghSlug`) — a naive `[^a-z0-9]` filter strips underscores and silently breaks snake_case jump-nav (it shipped once; caught only by rendering through GitHub's Markdown API). |
| Alerts (`> [!NOTE]`) | **Unreliable in wikis** — they render on READMEs/PRs but not consistently in the wiki renderer. Use a plain blockquote (`> **AI context:** …`) instead. |
| Mermaid | Renders natively in wikis via ` ```mermaid ` fences — use it for the architecture diagram (your domain model + request/data flow). |
| Collapsible | `<details><summary>…</summary></details>` works — wrap long prop tables or large examples to keep pages skimmable. |

### Reference-page quality (API-doc best practices)

- Lead each symbol with a one-line **business purpose**, not just mechanics ("creates the user *and sends the verification email*", not "`POST /users`").
- Per symbol: signature → params table → **example** → **failure modes**. Don't document only the happy path; for API routes, list status codes + error shapes (developers otherwise learn failures by trial and error).
- Consistent terminology and identical table shapes across pages — skimmability beats prose.
- Examples come from **real call sites** and live in the repo, so CI can catch drift (pair with `assess`).

Sources: [Diátaxis](https://diataxis.fr/start-here/) · [GitHub Docs — wiki sidebars/footers](https://docs.github.com/en/communities/documenting-your-project-with-wikis/creating-a-footer-or-sidebar-for-your-wiki) · [Gollum wiki (page-name slugging — but its `[[_TOC_]]` tag is NOT enabled on GitHub's hosted wiki)](https://github.com/gollum/gollum/wiki) · [GitHub — Mermaid in Markdown](https://github.blog/developer-skills/github-wiki/include-diagrams-markdown-files-mermaid/) · [Speakeasy — API doc best practices](https://www.speakeasy.com/api-design/documentation).

## Suggested Wiki page set

| Page | Source |
|---|---|
| `Home.md` | app summary + links to every page below (the landing page) |
| `Architecture.md` | your core domain model + key concepts + deployment (from `CLAUDE.md`/`AGENTS.md`, distilled). Include a ` ```mermaid ` diagram of the domain model + request flow. |
| `Reference-Components-*.md` | extract-docs.mjs + prose, split by area (one page per UI area) |
| `Reference-Hooks.md` | typedoc + prose |
| `Reference-API.md` | API routes (e.g. Express route dir) + prose |
| `Reference-Lib.md` | typedoc of `lib/` |
| `Database-Schema.md` | schema tables/columns (from `$CR_SCHEMA`) |
| `Getting-Started.md` | onboarding how-to: prose + stamped fact-blocks (gen-team-pages.mjs) — see "Team pages" |
| `Skill-Inventory.md` | team competency matrix: prose + stamped stack table (gen-team-pages.mjs) |
| `_Sidebar.md` | hand-built navigation (the default alpha sidebar is unhelpful); regenerate every publish |
| `_Footer.md` | common footer — "generated by /code-readability · last updated <date>" |

Keep pages **per-area, not one giant file** — the Wiki sidebar and search work better, and a model can load just the relevant page.

### Page Anatomy (data-flow / "how a screen works")

Reference pages document the API *surface*; a **Page Anatomy** page documents the *flow* — the round trip behind each screen (UI → hook → `api()` → route → ORM → DB → back) so a developer sees how a page works and where to change it. This is the Diátaxis *explanation/how-to* complement to the reference pages, and the highest-leverage doc for onboarding + throughput (a sequence diagram beats a wall of prose for "how does this feature work"). Backed by the living-docs pattern — [Mermaid sequence diagrams as code](https://github.com/mermaid-js/mermaid) that live next to the source and link into it.

Per screen, emit: a one-line **route / what / reads-writes**, a **Mermaid `sequenceDiagram`** of the *real* round trip, a **Layers & files** table (UI page · components · hooks · your API client (e.g. `src/lib/api.ts`) · routes · business logic · ORM/schema — every path a backtick repo path so `linkify-wiki.mjs` wires it to source), the **endpoints**, and a **"Where to make changes"** list mapping concrete tasks to exact files. Open the page with a single **shared-loop flowchart** (the 3-tier architecture) so the per-screen diagrams have a frame, and a "Jump to" anchor TOC.

**Generation — trace, don't guess.** One subagent per screen reads the page → its hooks → the `api()` endpoints → the Express routes → the business logic (`pure.ts` / views) → the `schema.ts` tables, and returns the filled template. (This is *derived from code*, not deterministic extraction like the reference pages — so re-verify against source when the data flow changes.)

**Mermaid gotchas (GitHub renderer):** in sequence message labels, no `{ }` (use parens) and **no `;`** (it's a statement separator — use `—` or `,`); `<br/>` is allowed in node/participant labels; never name a participant `end`; keep one `:` delimiter per message (extra colons in the text are fine). Validate by POSTing the page to the GitHub `/markdown` API and confirming each ```mermaid block produced a render hook — a parse error renders as an "Unable to render rich display" box. The `linkify` step skips fenced blocks, so diagram labels stay plain — put the linkable file paths in the tables/prose, not the diagram.

### Team pages (onboarding + skill inventory)

The Reference and Page-Anatomy pages document the *code*. Two pages document the *team* — written for humans, not derived from symbols:

- **`Getting-Started.md`** (Diátaxis *how-to*) — fresh `git clone` → running localhost: prerequisites, env config, dev commands, troubleshooting.
- **`Skill-Inventory.md`** (Diátaxis *explanation*) — a competency matrix (technology, design patterns, database, frontend, Agile/kanban) with proficiency targets (Aware / Working / Fluent) and where to learn each in the repo.

These are **hand-authored prose with stamped fact-blocks** — the inverse of the fully-generated reference pages. `scripts/gen-team-pages.mjs` owns four marker regions (the same mechanism as `stamp-codehealth.mjs`):

| Marker | Source | Page |
|---|---|---|
| `<!--cr:prereqs-->` | `engines.node` / `.nvmrc` + `packageManager` | Getting-Started |
| `<!--cr:env-->` | parsed `.env.example` (var · example · grouping comment) | Getting-Started |
| `<!--cr:scripts-->` | every `npm run` target in the root `package.json` | Getting-Started |
| `<!--cr:stack-->` | signal deps + versions scanned across all `package.json` | Skill-Inventory |

```bash
# First time: scaffold starter pages (prose template + markers), then fill the prose.
node .claude/skills/code-readability/scripts/gen-team-pages.mjs /tmp/cr-wiki --scaffold
# Every publish after: regenerate only the cr: fact-blocks; prose is preserved (idempotent).
node .claude/skills/code-readability/scripts/gen-team-pages.mjs /tmp/cr-wiki
```

Config (env, defaults work in a repo root): `CR_PKG` (root package.json), `CR_ENV_EXAMPLE` (`.env.example`), `CR_REPO_ROOT` (dir to scan for stack package.json files). Run the stamp **before** `linkify-wiki` so the file/script mentions in the new blocks get source links. The prose (accounts, ports, troubleshooting, the matrix rows) is project-specific — the scaffold ships `<!-- TODO -->` placeholders you fill once.

## Publishing (SSH, mirrors `/github-wiki wiki`)

```bash
ssh -T git@github.com 2>&1 | head -1                 # verify auth
WIKI_URL="git@github.com:<owner>/<repo>.wiki.git"     # origin .git → .wiki.git
git clone "$WIKI_URL" /tmp/cr-wiki 2>/dev/null || git -C /tmp/cr-wiki pull
cp /tmp/cr-docs/pages/*.md /tmp/cr-wiki/
# Cross-reference: link every backtick file/script mention to its source on GitHub.
node .claude/skills/code-readability/scripts/linkify-wiki.mjs <repo-root> /tmp/cr-wiki \
  https://github.com/<owner>/<repo>/blob/main
git -C /tmp/cr-wiki add .
git -C /tmp/cr-wiki commit -m "docs: regenerate API reference (/code-readability)"
git -C /tmp/cr-wiki push origin master                # wiki has one master branch, no PRs
```

**Guardrails:**
- Wiki **push is SSH-only** — fine-grained PATs get 403 (same quirk the `/github-wiki wiki` skill documents).
- **Marker check:** every generated page leads with `<!-- generated by /code-readability — edits will be overwritten -->`. Before overwriting an existing same-named page, confirm it has the marker; if not, it's hand-authored — surface it and ask, don't clobber.
- **Cross-reference always:** run `scripts/linkify-wiki.mjs` (above) as the last step before commit — it wraps every backtick-quoted reference to a real tracked file (`src/lib/foo.ts`, `schema.ts`, a `scripts/*.mjs`) in a link to that file on GitHub, so docs are one click from the code. Safe + idempotent (only links real files, skips code fences / existing links / commands / type names / DB table names). Private repos: the blob links resolve for the same authenticated users who read the wiki.
- Page links use Wiki `[[Page]]` / `[[Page#anchor]]` syntax, not repo-relative paths.
- Always `git pull` an existing clone before writing (stale-clone trap).

### Generating docs for non-TypeScript frontends
The tool layer above (`react-docgen-typescript` + TypeDoc) is for TS/React. When a project ships hand-written HTML/CSS/Sass/JS, swap in the matching generator (the comment standards are in `references/comment-style.md` → "Beyond TypeScript"):
- **Vanilla JS** → `jsdoc` or `documentation.js` (JSDoc → HTML/Markdown API site).
- **CSS / Sass** → **KSS** (kss-node) for a *living style guide* of components+states, **SassDoc** for the design-system API.
- **Components (any framework)** → **Storybook autodocs** or **React Styleguidist** turn JSDoc/TSDoc + stories into living docs that render the real UI.
Then feed their Markdown through the same prose-layer + linkify + publish flow.

## Drift control

Because the tool layer regenerates from comments, stale docs mean stale comments. The recommended guard is a small CI check (a `docs:check` script) that fails when any exported declaration in the documented areas loses its adjacent `/** */` TSDoc — bare re-exports excluded. Configure the covered areas as a list of source dirs (your `<your source dirs>`) and run it as a dependency-free job in CI, parallel to lint/type-check/test/build. (The reference project implemented this as `scripts/check-doc-coverage.mjs`; it isn't shipped with the skill — add an equivalent to your repo.)

- The check should require the TSDoc to sit **directly above** the export (no blank line) — the same adjacency react-docgen-typescript needs, so it doubles as a placement guard.
- To extend coverage to a new area, add its directory to the covered-dirs list.
- A `--list` flag that prints every export with ✓/✗ helps triage.

The **Team pages** use a different drift guard: their factual blocks are *stamped from the repo* on every publish (`gen-team-pages.mjs`), so a changed `npm run` script, env var, Node version, or dependency version flows into the page automatically — only the surrounding prose is hand-maintained. Wire the stamp into the same weekly job that runs `codehealth:report` + `wiki:stamp` so onboarding never lists a command that no longer exists.
